## Divergência ROADMAP original ↔ execução real

O plano original (na seção abaixo) tinha Phases 3-7 dedicadas a "Portal do Cliente" e "Painel Admin" como features-end-to-end. A execução real pivotou: criou primeiro uma camada de compatibilidade (`core/`), depois desmembrou o monolito admin em actions modulares.

**Resultado:** O escopo dos requirements foi entregue (todas as 32 requirements satisfeitas), mas com agrupamento diferente nas phases. STATE.md documentou a realidade.

---

## Key accomplishments

1. **Stack moderna em PHP vanilla** — `area-cliente/core/` com Auth, Csrf, Database singleton, Migrations, Logger, Upload, Processo. Sem framework, mantendo compatibilidade Hostinger shared.
2. **17 actions modulares** — POST handlers do admin extraídos de `processamento.php` monolítico para `actions/admin/cliente_*.php`, `financeiro_*.php`, `pendencia_*.php`, `entregavel_*.php`, etc.
3. **Deploy automático CI/CD** — GitHub Actions com FTPS para Hostinger, gerando `db_credentials.php` a partir de secrets a cada push. Resolveu o caos de credenciais expostas no repo.
4. **Segurança hardened** — credenciais fora do repo, .htaccess bloqueando `core/`, `config/`, `maintenance/`, scripts de debug removidos, senha admin migrada para .env.
5. **Landing mobile-first** — hero com gradiente da marca, FAB "Área do Cliente", 5 serviços com cards, seção "Como Funciona" em 3 passos, footer com redes sociais e Instagram.
6. **Portal do cliente operacional** — login dual (admin + cliente), timeline de 9 etapas, dashboard com pendências e financeiro, upload de comprovantes, responsivo mobile.

## Dívidas técnicas levadas para Milestone v2.0

Identificadas durante debug session [admin-modificar-cliente.md](../debug/admin-modificar-cliente.md):

- **D1:** Validação CSRF em todos `actions/admin/*.php` está mal escrita: `if (isset($_POST['csrf_token']) && !Csrf::validateToken(...))` → não rejeita requests sem token. Correto: `if (empty($_POST['csrf_token']) || !Csrf::validateToken(...))`.
- **D2:** `cliente_impersonate.php:9` usa `require '../../db.php'` (path relativo, frágil). Trocar para `require_once __DIR__ . '/../../includes/init.php'`.
- **D3:** `includes/processamento.php` legado coexiste com `actions/admin/*.php` (duplicação de lógica). Auditar e remover.
- **D4:** `admin.php:62` usa JS para redirect em vez de `header()`.

Adicionais identificadas hoje:
- **D5:** GitHub Actions secrets foram vazados em chat (rotacionar todas as senhas — SSH, FTP, MySQL, ADMIN_PASSWORD).
- **D6:** `.git` está em `C:\Users\diego\` (HOME do usuário) por inicialização errada. Realocar para pasta de trabalho dedicada.
- **D7:** Repo `applouvor` (App Louvor) tem outros paths absolutos potencialmente quebrados; auditar JS files.
